## Phase 1: Requirements Definition (30 minutes)

This is the conversational phase where we work with Claude to define exactly what needs to be built. According to [The Ralph Playbook](https://claytonfarr.github.io/ralph-playbook/), you should spend 30+ minutes in this phase.

### The Conversation

**You:**
> "I want to build a REST API for managing todos with user authentication. Let's define the requirements together."

**Claude:**
> "Great! Let's break this down systematically. First, what core functionality do you need?

> 1. User management (registration, login)?
> 2. Todo CRUD operations?
> 3. Any filtering or pagination?
> 4. What about sharing or collaboration features?"

**You:**
> "Let's keep it focused:
> - User registration and login with JWT
> - Each user can only see their own todos
> - Full CRUD for todos (create, read, update, delete)

**Claude:**
> "Perfect. Let me create a specs file for this. What about the tech stack?"

**You:**

**Claude:**
> "Excellent choices. A few more questions:
>
> 1. Password requirements? (min length, complexity?)
> 2. JWT expiry time?
> 3. API versioning? (/v1/...)?
> 4. Error response format?"

**You:**
> "Passwords: min 8 chars, at least one number. JWT: 7 days. Yes to versioning with /api/v1. Standard REST error format with status codes."

**Claude:**
> "Got it. Let me create comprehensive specs..."

### Generated Specs Files

Claude creates `specs/authentication.md`:

```markdown
# Authentication Specification

## Overview
JWT-based authentication with httpOnly cookies for security.

## User Registration

**Endpoint:** POST /api/v1/auth/register

**Request Body:**
```json
{
  "email": "user@example.com",
  "password": "password123",
  "name": "John Doe"
}
```

**Validation Rules:**
- Email: valid email format, unique
- Password: min 8 characters, at least 1 number
- Name: 2-50 characters

**Success Response:** 201 Created
```json
{
  "user": {
    "id": "uuid",
    "email": "user@example.com",
    "name": "John Doe"
  },
  "token": "jwt-token"
}
```

## User Login

**Endpoint:** POST /api/v1/auth/login

**Request Body:**
```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

**Success Response:** 200 OK
- Sets httpOnly cookie with JWT
- Returns user object

**Error Responses:**
- 401: Invalid credentials
- 400: Validation errors

## Authentication Flow

1. User registers or logs in
2. Server generates JWT (expires in 7 days)
3. JWT stored in httpOnly cookie (secure, sameSite: strict)
4. Protected routes verify JWT via middleware
5. User ID extracted from JWT for authorization
```

And `specs/todos.md`:

```markdown
# Todo API Specification

## Data Model

```typescript
interface Todo {
  id: string;
  title: string;
  description: string | null;
  completed: boolean;
  dueDate: Date | null;
  userId: string;
  createdAt: Date;
  updatedAt: Date;
}
```

## Endpoints

### Create Todo
POST /api/v1/todos

**Authentication:** Required

**Request:**
```json
{
  "title": "Buy groceries",
  "description": "Milk, eggs, bread",
  "dueDate": "2026-01-20T10:00:00Z"
}
```

**Validation:**
- title: required, 1-200 characters
- description: optional, max 1000 characters
- dueDate: optional, must be future date
- completed: defaults to false

### List User's Todos
GET /api/v1/todos

**Query Parameters:**
- completed: boolean (filter)
- sortBy: createdAt | dueDate | title
- order: asc | desc

### Get Single Todo
GET /api/v1/todos/:id

**Authorization:** User must own the todo

### Update Todo
PUT /api/v1/todos/:id

**Authorization:** User must own the todo

### Delete Todo
DELETE /api/v1/todos/:id

**Authorization:** User must own the todo

## Business Rules

1. Users can only access their own todos
2. Cannot modify another user's todos
3. Deleting a user cascades to their todos
4. All dates stored in UTC
```

## Handling Errors (What Went Wrong)

Ralph hit a few snags along the way. Here's how they were resolved:

### Error 1: Test Database Configuration (Iteration 6)

**Error:**
```
[Iteration 6] Running tests...
Error: P1001: Can't reach database server at localhost:5432
```

**What happened:** Ralph tried to run tests but the test database wasn't configured.

**How Ralph fixed it:**
```
[Iteration 7] Detected database connection issue
[Iteration 7] Updating test script to use TEST_DATABASE_URL
[Iteration 7] Added setup script to create test database
[Iteration 7] Tests now passing
```

**Learning:** Ralph read the error, understood the issue, and fixed it autonomously by adding proper test database configuration.

### Error 2: TypeScript Type Error (Iteration 14)

**Error:**
```
[Iteration 14] Type checking...
Error: Property 'userId' does not exist on type 'Request'
```

**What happened:** The Express Request type didn't include our custom `userId` property from auth middleware.

**How Ralph fixed it:**
```
[Iteration 15] Creating types/express.d.ts
[Iteration 15] Extending Express Request interface
[Iteration 15] Type check passing
```

**Solution Ralph implemented:**

```typescript
// types/express.d.ts
declare namespace Express {
  export interface Request {
    userId?: string;
  }
}
```

**Error:**

[Iteration 22] Test failed: "should filter completed todos"
Expected 2 but got 0

**Learning:** Ralph debugged the async issue and fixed the test properly.

## Key Lessons Learned

### What Worked Well

1. **Clear specs = fast implementation**: The 30-minute conversation upfront saved hours of back-and-forth
2. **Atomic tasks**: Breaking the project into 12 tasks allowed Ralph to complete each in 2-3 iterations
3. **Test-driven backpressure**: Requiring tests to pass prevented broken code from propagating
4. **Progress tracking**: progress.txt provided valuable context across iterations
5. **Fresh context**: Each iteration started clean, preventing context degradation

### What Could Be Improved

1. **Test database config**: Should have been in the plan explicitly
2. **TypeScript types**: Adding custom Express types to the plan would have prevented the error
3. **Async testing patterns**: More explicit test patterns in specs would help
